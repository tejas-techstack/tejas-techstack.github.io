Introduction to Compilers

Definition

A compiler is a program that translates source code written in a high-level language into equivalent target code (typically machine code or assembly), while reporting errors.

Compiler

A compiler translates the entire source program written in a high-level language into equivalent target code (machine code or another language) before execution. The translation happens all at once, and a separate executable is produced.

Interpreter

An interpreter reads and executes the source program statement-by-statement, without generating a complete machine-code file first.

CompilerInterpreter
Translates whole program at onceExecutes one statement at a time
Faster execution (after compilation)Slower execution (translates on the fly)
Error reported after scanning all codeError reported immediately at that line
Produces object code / executableNo separate object file produced
E.g., C, C++, Java (to bytecode)E.g., Python, Ruby, JavaScript (interpreted)

Two Types of Compilers

  • Cross Compiler: Runs on one machine (host) but generates code for a different machine (target). E.g., compiling ARM code on an x86 machine. Useful in embedded development.
  • Just-In-Time (JIT) Compiler: Compiles bytecode (e.g., Java bytecode) into native machine code at runtime, combining benefits of both compilation and interpretation. Used in JVM, .NET CLR.
Other types include: Bootstrapping compiler (compiler written in its own language), Single-pass compiler (one scan), Multi-pass compiler (multiple scans for better optimization).

Front End

The front end is the language-dependent part of the compiler. It analyzes the source program and checks its correctness. It comprises:

  • Lexical Analysis – tokenizes source code
  • Syntax Analysis – verifies grammar structure
  • Semantic Analysis – type checking, scope checking
  • Intermediate Code Generation – produces IR (e.g., 3-address code)

The front end produces an Intermediate Representation (IR) which is the output passed to the back end.

Back End

The back end is the machine-dependent part. It transforms the IR into efficient target code. It comprises:

  • Code Optimization – improves IR for speed/space
  • Code Generation – maps IR to target instructions, allocates registers

Advantage of Separation

By using an IR as a bridge, you can create m front ends (one per source language) and n back ends (one per target machine), giving m×n compilers with only m+n components.

Lex

Lex (Lexical Analyzer Generator) is a tool that takes a specification file consisting of regular expressions (patterns) and associated actions, and automatically generates C code for a lexical analyzer. The generated scanner (lexer) reads input characters, matches them against the patterns, and executes the corresponding action (e.g., return a token).

Lex Program Structure

%{ 
  /* C declarations */
%}
/* Definitions */
%%
/* Rules: pattern  { action } */
[a-zA-Z][a-zA-Z0-9]*  { return ID; }
[0-9]+                { return NUM; }
%%
/* User C code */
  • Section 1 (before first %%): Declarations and definitions
  • Section 2 (between %%): Pattern-action rules
  • Section 3 (after second %%): Auxiliary C functions

Yacc

Yacc (Yet Another Compiler Compiler) is a parser generator. It takes a grammar specification (context-free grammar with associated semantic actions) and generates a LALR(1) parser in C. Yacc is typically used together with Lex — Lex provides tokens, Yacc uses them to parse and apply semantic actions.

Modern equivalents: Flex (replacement for Lex) and Bison (GNU replacement for Yacc).

Role of a Lexer (Scanner)

The lexer (lexical analyzer) is the first phase of a compiler. It reads the raw character stream of the source program and groups characters into meaningful sequences called lexemes, returning a corresponding token (a <token-name, attribute-value> pair) to the parser on demand. It also:

  • Strips whitespace and comments
  • Manages the input buffer efficiently
  • Interacts with the symbol table

Common Lexer — Is It Possible?

No, a single common lexer for all languages is generally not possible because different languages have fundamentally different token sets, keyword lists, and lexical rules.

  • Example 1 — Keywords: In C, int is a keyword (type). In Python, there is no int keyword (it's a built-in function). A common lexer cannot know which identifiers are reserved words without being language-specific.
  • Example 2 — String literals: In C, strings use double quotes "hello" and characters use single quotes 'a'. In Python, both 'hello' and "hello" are valid strings, and triple quotes """...""" are also valid. The lexical rules differ significantly.
  • Example 3 — Comments: C uses /* */ and //; Python uses #; Haskell uses -- and {- -}. These cannot be handled by a single rule set.
Conclusion: Each language has its own token specification, so a separate lexer (or parameterized lexer) is needed per language.

Lexer–Parser Interaction

In most languages the lexer can identify tokens independently, but there are cases where context from the parser is needed:

Example — typedef in C: Consider the following C code:

typedef int MyInt;
MyInt x;

When the lexer encounters MyInt in the second line, it matches as an identifier lexeme. However, it should be treated as a type name (like int) — this is only known after the parser has processed the typedef declaration and updated the symbol table. So the lexer needs to consult the symbol table (fed by the parser/semantic phase) to return the correct token TYPE_NAME instead of ID.

Another example — Fortran: In older Fortran, whitespace is insignificant. The statement DO 5 I = 1,25 is a loop, but DO 5 I = 1.25 is an assignment to variable DO5I. The lexer cannot distinguish them without parser context.

Lexical Analysis

Why Buffering?

Reading source code one character at a time with a system call per character is extremely slow. Buffering loads a large block of characters into memory at once, making character access an in-memory operation. Additionally, lexical analysis requires lookahead — the scanner must peek at the next character(s) to decide whether a token has ended (e.g., >= vs >), so a buffer that supports both forward reading and retraction is needed.

Buffering Methods

1. One-Buffer Scheme: A single buffer of size N (e.g., 4096 bytes) is filled from the input. Two pointers — lexemeBegin (start of current token) and forward (current scan position) — traverse the buffer. On exhaustion, the buffer is refilled. Problem: a token cannot span a buffer boundary easily.

2. Two-Buffer (Pair of Buffers) Scheme: Two buffers of equal size N are used alternately. When forward reaches the end of the first buffer, the second is loaded and scanning continues seamlessly. This allows tokens to span the boundary between fills.

Sentinels

A special character eof is placed at the end of each buffer half. When forward encounters eof, the scanner knows the buffer is exhausted and loads the next half. This eliminates the need for an explicit bounds check on every character read — only the eof check is needed. Advantage: reduces the number of tests from 2 per character (end-of-buffer + special-character) to just 1 (check for eof/special char only).

Input Buffering

Input buffering is a technique where instead of reading one character at a time from the source file (which requires a system call per character — very slow), a large block of characters is loaded into an in-memory buffer at once. The lexer then reads from this fast memory buffer.

Why Necessary

  1. Performance: System calls are expensive; buffering amortizes the cost across many character reads.
  2. Lookahead: The scanner needs to look ahead (sometimes several characters) to determine if the current lexeme has ended. For example, to recognize != vs !, or to distinguish while (keyword) from whileloop (identifier).
  3. Retraction: After reading one character too far (to confirm end of a token), the scanner must "put back" the character. Buffers support this via a pointer decrement.

Advantage of Sentinels

Without sentinels, for each character read we need two checks:

  1. Is forward at the end of the buffer? (buffer boundary check)
  2. Is the character a special character (e.g., newline, whitespace)?

With a sentinel (the special character eof placed at position N+1 of each half), both checks reduce to one check: test if the current character is eof. If it is, we decide whether it is genuine end-of-input or just end-of-buffer-half and reload. This halves the number of conditional tests during lexical scanning, significantly improving performance for large programs.

Without sentinel: if (forward == end_of_buf || isSpecial(ch)) ...  // 2 checks
With sentinel:    if (ch == EOF) ...                               // 1 check

Lexical Errors

A lexical error occurs when the scanner cannot match the current sequence of characters to any valid token pattern for the language. Examples:

  • An illegal character in the source: @ in C
  • An unterminated string literal: "hello without closing quote
  • An invalid numeric literal: 1.2.3
  • Excessively long identifier (beyond symbol table limit)

Recovery strategies: delete the offending character, insert a missing character, replace a character, or transpose adjacent characters.

A transition diagram is a visual representation of a finite automaton used to specify how the scanner recognizes tokens. Each node represents a state (stage of token recognition), edges (arrows) represent transitions triggered by input characters, and accepting states (double circles) indicate successful token recognition.

Purpose:

  • Provides a clear blueprint for implementing the scanner as a set of if/switch statements or a table-driven DFA.
  • Directly derived from the regular expressions for each token class — regex → NFA → DFA → transition diagram.
  • Handles lookahead and retraction explicitly (marked on transitions).
  • Multiple transition diagrams (one per token class) are composed into a single lexer that tries each in sequence.

Regular Definition

A regular definition is a way to give names to regular expressions so they can be referenced in subsequent regular expressions, making complex patterns more readable and modular. The form is:

d₁ → r₁
d₂ → r₂
...
dₙ → rₙ

where each dᵢ is a distinct name and rᵢ is a regular expression over the alphabet plus the previously defined names d₁, …, dᵢ₋₁.

Example — C Identifiers and Integer Literals

letter_  → [A-Za-z_]
digit    → [0-9]
id       → letter_ (letter_ | digit)*
integer  → digit+

Here id is defined using the previously defined names letter_ and digit.

Purpose

  • Readability: Breaking a complex regex into named parts is much easier to understand.
  • Reuse: Sub-patterns like digit can be referenced multiple times without repeating [0-9].
  • Maintainability: Changing a sub-pattern in one place updates all references.

Design Overview

A lexical analyzer can be designed in two main ways: hand-coded (writing a DFA manually) or generated (using a tool like Lex). The diagram below shows the interaction between a lexer and parser.

How It Works

  1. Source code is loaded into the input buffer (two-buffer scheme).
  2. The lexer uses a DFA (or simulated NFA) driven by transition tables/diagrams to recognize patterns.
  3. When a pattern is matched, the lexer returns a token to the parser and inserts the lexeme into the symbol table if it is an identifier or literal.
  4. The parser calls getNextToken() whenever it needs the next token.

Pattern Meanings

  • ab? matches: a or ab
  • ab?b* matches: a, ab, abb, abbb, …
  • bc matches: bc

Lex Ambiguity Resolution Rules

  1. Longest match (maximal munch): Among all patterns that match a prefix of the remaining input, choose the one that matches the longest string.
  2. Rule priority: If two patterns match the same longest string, choose the pattern listed first in the specification.

Method 1: Install Keywords in Symbol Table Initially

All keywords are pre-loaded into the symbol table before scanning begins. When the lexer matches a lexeme as an identifier pattern, it looks it up in the symbol table. If found as a keyword, it returns the keyword token; otherwise it returns an ID token.

// Pre-load
symtable.insert("while", WHILE);
symtable.insert("for",   FOR);
// During scanning:
lexeme = matchIdentifier();
token = symtable.lookup(lexeme);
if (token == null) return ID;
else return token;

Method 2: Separate Patterns for Keywords

Each keyword is given its own pattern in the Lex specification, listed before the general identifier pattern. Due to the first-rule-wins tie-breaking rule, while will match the keyword pattern (not the identifier pattern) because keywords are listed first.

%%
while    { return WHILE; }
for      { return FOR; }
[a-zA-Z][a-zA-Z0-9]*  { return ID; }  /* listed last */
Method 1 is preferred in practice as it requires only one pattern for all identifiers and keywords.

Syntax Error Handling & Recovery

Function of Error Handler

When a parser encounters an error (input doesn't match the expected grammar), the error handler must:

  1. Report the error with a clear, useful message (location, nature).
  2. Recover from the error so parsing can continue and detect subsequent errors.
  3. Not slow down processing of correct programs.

Error Recovery Strategies

1. Panic Mode: The parser discards input tokens one at a time until it finds a token from a designated set of synchronizing tokens (e.g., statement terminators like ;, block delimiters like }). Then parsing resumes from that point. Simple and guaranteed to terminate but may skip large portions of input.

2. Phrase-Level (Statement Mode) Recovery: The parser makes a local correction — inserting, deleting, or replacing a token — to fix the error at the point where it was detected. For example, if a semicolon is expected but not found, insert one. Risk: may introduce infinite loops if the correction doesn't advance the input.

3. Error Productions: Common erroneous constructs are added to the grammar as error productions. When such a production is used, a diagnostic is issued. This anticipates common mistakes (e.g., 5 + * 3 as a common typo).

4. Global Correction: The parser finds the theoretically minimum number of changes (insertions, deletions, replacements) to the input to make it syntactically correct. Computationally expensive; rarely used in practice.

Panic Mode Recovery

Idea: On encountering an error, the parser panics and discards symbols from the input stream (or pops stack symbols) until it finds a token that can begin a valid phrase — a synchronizing token.

Synchronizing tokens are typically: ;, }, begin, end, if, else — tokens that clearly delimit statements or blocks.

Algorithm:

  1. Detect error (no valid production for current input token).
  2. Report error message.
  3. Discard input tokens until a synchronizing token is found.
  4. Resume parsing from that token.

Example: In { a = 1 b = 2; } (missing semicolon after first statement), on error the parser discards tokens until ; or }, skipping b = 2 but correctly parsing the rest.

Pros: Simple, always terminates. Cons: May miss detecting further errors in the skipped input.

Phrase-Level Recovery

Idea: Make the smallest local change to the input to allow parsing to continue. Common corrections include:

  • Insert a missing token (e.g., insert ; when expected)
  • Delete an erroneous token (e.g., delete extra ))
  • Replace one token with another

Example: Parser expects ) but finds ;. Phrase-level: replace ; with ) and continue.

Implementation in LL parsers: When M[A, a] is error, the routine for A's error entry either pops A from the stack (deleting a nonterminal), skips the current token, or pushes a new token.

Pros: Can detect more errors per parse. Cons: Risk of entering infinite loop if correction doesn't consume input; requires careful design.

See Q27 and Q28 for full details. Brief summary of two strategies:

1. Panic Mode

Parser discards input tokens until a synchronizing token (e.g., ;, }) is found. Simple, guaranteed to terminate, but may skip valid code. Best for recovering from serious errors like missing delimiters.

2. Error Productions

The grammar is augmented with extra productions that recognize common erroneous constructs. When such a production reduces during parsing, a warning/error message is printed. For example, adding a production expr → expr + + term to catch the common mistake of typing ++ between operands.

Advantage: Provides very targeted, informative error messages for known mistakes. Disadvantage: Requires advance knowledge of common errors; grammar becomes larger.

(i) Left-Recursive Grammar

A grammar is left-recursive if there exists a nonterminal A such that A ⇒+ Aα for some string α. Direct left recursion example:

E → E + T | T

Here E → E + T is left-recursive because E appears as the leftmost symbol on the RHS. Left recursion causes top-down parsers (like LL) to loop infinitely.

(ii) Leftmost Derivation

In a leftmost derivation, at each step the leftmost nonterminal in the sentential form is replaced. Example with grammar E → E+T | T, T → id:

E ⇒ E+T ⇒ T+T ⇒ id+T ⇒ id+id

Each step replaces the leftmost nonterminal. Top-down parsers generate leftmost derivations.

(iii) Ambiguous Grammar

A grammar is ambiguous if there exists a string that has more than one parse tree (or equivalently, more than one leftmost derivation). Example:

E → E + E | E * E | id

The string id + id * id has two parse trees: one where + is the root, another where * is the root. Ambiguous grammars are problematic for compilers as the meaning of a program would be undefined.

(iv) LR(0) Item

An LR(0) item is a production with a dot (•) at some position in the right-hand side, indicating how much of the production has been seen so far. Example for production A → XYZ:

A → •XYZ   (nothing seen yet)
A → X•YZ   (X has been seen)
A → XY•Z   (XY seen)
A → XYZ•   (complete: ready to reduce)

Items are used to construct the LR(0) automaton (collection of canonical LR(0) items) which drives the LR parser.

Recursive-Descent Parsing

RDP is a top-down parsing technique where we write a set of mutually recursive procedures, one per nonterminal. Each procedure tries to match its corresponding grammar production against the current input.

// For production S → +SS | *SS | a
void S() {
  if (current_token == '+') {
    match('+'); S(); S();
  } else if (current_token == '*') {
    match('*'); S(); S();
  } else if (current_token == 'a') {
    match('a');
  } else error();
}

Key points:

  • Works directly on LL(1) grammars (or those that can be made LL(1)).
  • When there is a choice (alternative productions), it uses the lookahead token to decide which alternative to take.
  • Simple to implement by hand but requires the grammar to have no left recursion and no common prefixes.
  • Backtracking RDP can handle more grammars but is less efficient.

Canonical LR(1) Items

An LR(1) item is an LR(0) item augmented with a lookahead symbol: a pair [A→α•β, a] where 'a' is a terminal (the lookahead). The lookahead 'a' indicates that a reduction A→αβ should be made only when the next input symbol is 'a'.

Example: [S → A • B, c]
  Meaning: we've seen A, expecting to see B; 
  if β is empty and next input is 'c', reduce.

The CLOSURE of LR(1) items: if [A→α•Bβ, a] is in the set and B→γ is a production, then add [B→•γ, b] for each b ∈ FIRST(βa).

The GOTO function moves the dot past a grammar symbol.

Why CLR(1) is Most Powerful

  1. Precise lookahead: Each item carries the exact set of terminals that justify a reduce action. This is more precise than SLR(1) (which uses FOLLOW) and LALR(1) (which uses merged lookaheads).
  2. Resolves more conflicts: Situations where FOLLOW or merged lookaheads create spurious conflicts are handled correctly because the lookahead is context-specific.
  3. Recognizes all LR(1) grammars: Any deterministic context-free language has an LR(1) grammar, and CLR(1) can parse all of them without conflict.
  4. Subsumes all other LR parsers: Every grammar parseable by LR(0), SLR, or LALR is also parseable by CLR(1).

Downside: The number of states can be very large (exponentially more than LALR). Yacc uses LALR as the practical compromise.

LR Parser Model

Inherited Attributes

An attribute is inherited if its value at a node in the parse tree is computed from the attribute values of the node's parent or siblings (i.e., passed down the tree). Example:

D → T L       { L.type = T.type }
T → int       { T.type = integer }
L → L₁ , id  { L₁.type = L.type; addtype(id, L.type) }
L → id        { addtype(id, L.type) }

Here L.type is an inherited attribute — its value flows from the parent D down to the children L.

L-attributed SDT

An SDD is L-attributed if each inherited attribute of a symbol on the right-hand side of a production depends only on:

  1. Inherited attributes of the nonterminal on the left-hand side, OR
  2. Inherited or synthesized attributes of symbols to the left of that symbol in the production.

L-attributed SDDs can be evaluated in a single left-to-right traversal, making them suitable for top-down (LL) parsers.
