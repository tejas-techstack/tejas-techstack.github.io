§ 4.1 — Intermediate Representations: Quadruples, Triples, SSA
Q4.1 Write short notes on: (i) Quadruples and (ii) Static Single Assignment (SSA)
5 marks

i) QUADRUPLES

A quadruple is a four-field record used to represent three-address code. Each instruction has the form: (op, arg1, arg2, result).

FieldMeaning
opThe operator (e.g., +, *, uminus, =, param, call, if)
arg1First operand (or the only operand for unary ops)
arg2Second operand (empty for unary ops)
resultWhere the result is stored (temp variable or variable)

Example: For a = b * -c + b * -c:

  #   op       arg1   arg2   result
  0   uminus   c             t1
  1   *        b      t1     t2
  2   uminus   c             t3
  3   *        b      t3     t4
  4   +        t2     t4     t5
  5   =        t5            a

Disadvantage: Quadruples use a large number of temporary variables. Renaming temporaries is expensive. Every three-address instruction requires a separate record.

ii) STATIC SINGLE ASSIGNMENT (SSA)

SSA is an intermediate representation in which every variable is defined (assigned) exactly once. When a variable is assigned multiple times in the original code, it is renamed to a new version (subscript) each time.

Key Property: Each use of a variable refers to exactly one definition. This simplifies many data-flow analyses.

φ-function (Phi function): At control-flow merge points (e.g., after if-else), a special φ-function merges versions:

  Original code:           SSA form:
  x = 1                    x₁ = 1
  if (cond) x = 2          if (cond) x₂ = 2
  y = x + 1                x₃ = φ(x₁, x₂)
                           y = x₃ + 1

Benefits of SSA:

  • Simplifies constant propagation and dead code elimination
  • Makes def-use chains explicit and compact
  • Enables efficient register allocation
  • Facilitates common subexpression elimination

SSA is widely used in modern compilers (GCC, LLVM).

Q4.2 Write short notes on: (i) Triples and (ii) Static Single Assignment (SSA) [Dec 2023 / Jul 2023]
5 marks

i) TRIPLES

A triple is a three-field record (op, arg1, arg2). Unlike quadruples, there is no explicit result field; instead, the result is implicitly the position (index) of the instruction in the array.

#oparg1arg2
0uminusc—
1*b(0)
2uminusc—
3*b(2)
4+(1)(3)
5=a(4)

Advantage over Quadruples: No need to create temporary variable names; saves memory.

Disadvantage: References are by position. If instructions are reordered (e.g., during optimization), all position references must be updated. This makes code movement difficult.

Indirect Triples: To overcome the reordering problem, an auxiliary array of pointers to triples is used. Reordering the auxiliary array doesn't require changing the triples themselves.

ii) SSA — See Q4.1 above for full explanation

SSA assigns unique names to each definition of a variable, enabling cleaner data-flow analysis. φ-functions handle merges at join points in the control-flow graph.

5 marks

STEP 1: Original Code

  x = u - t
  y = x * v
  if (y > 100)
      x = y + w
  else
      x = y - w
  y = t - z
  y = x * y

STEP 2: SSA Form

Rename each assignment to a fresh version. At the merge point after the if-else, use a φ-function for x since it is defined in both branches.

  x₁ = u - t
  y₁ = x₁ * v
  if (y₁ > 100)
  [then branch]:  x₂ = y₁ + w
  [else branch]:  x₃ = y₁ - w

  x₄ = φ(x₂, x₃)   ← merge point: x from then or else

  y₂ = t - z
  y₃ = x₄ * y₂
Notes:
  • x is defined in both branches → needs φ-function after the if-else
  • y after the if-else: the last assignment is y = t-z (not affected by the branch), then y = x*y uses that
  • Variables u, t, v, w, z are assumed to be defined before this block (as u₁, t₁, etc.)

§ 4.3 — Code Optimization Techniques
Q4.11 Optimize the code: int a=30; int b=9-(a/5); int c; c=b*4; if(c>10){c=c-10;} return c*(60/a); using Constant Folding, Constant Propagation, and Dead Code Elimination. [Appears in multiple papers]
5 marks

ORIGINAL CODE

  int a = 30;
  int b = 9 - (a / 5);
  int c;
  c = b * 4;
  if (c > 10) {
      c = c - 10;
  }
  return c * (60 / a);

STEP 1: CONSTANT FOLDING

Evaluate constant sub-expressions at compile time:

  • a / 5 → 30 / 5 = 6
  • 9 - 6 = 3 → b = 3
  • 60 / a = 60 / 30 = 2
  int a = 30;
  int b = 3;        // folded: 9-(30/5) = 3
  int c;
  c = b * 4;
  if (c > 10) {
      c = c - 10;
  }
  return c * 2;     // folded: 60/30 = 2

STEP 2: CONSTANT PROPAGATION

Substitute known constant values of variables:

  • b = 3, so b * 4 = 3 * 4 = 12 → c = 12
  int c;
  c = 12;           // propagated: b=3, 3*4=12
  if (c > 10) {     // 12 > 10 → TRUE (always)
      c = c - 10;   // c = 12 - 10 = 2
  }
  return c * 2;

STEP 3: DEAD CODE ELIMINATION

Since c = 12 > 10 is always true, the if-branch always executes. We can eliminate the conditional:

  • a and b are no longer referenced → remove them
  • c = 12 - 10 = 2 can be folded
  c = 2;
  return c * 2;
  // → return 4;

FINAL OPTIMIZED CODE

  return 4;
Summary: Constant folding reduced expressions; constant propagation substituted known values; dead code elimination removed the unreachable else-branch and unused variables. The result is a single return 4.
Q4.12 Explain the following machine-independent code optimization techniques with examples: (1) Constant Folding and Constant Propagation, (2) Common Subexpression Elimination, (3) Dead Code Elimination [Dec 2024]
6 marks

1. CONSTANT FOLDING AND CONSTANT PROPAGATION

Constant Folding: Evaluate constant expressions at compile time rather than at runtime.

  Before:  x = 3 * 4 + 2;   y = z * 0;
  After:   x = 14;           y = 0;

Constant Propagation: If a variable is known to hold a constant value, replace its uses with that constant.

  Before:  a = 5;   b = a + 3;   c = a * b;
  After:   a = 5;   b = 8;       c = 40;
  // a and b can then be eliminated if unused

2. COMMON SUBEXPRESSION ELIMINATION (CSE)

If the same expression is computed more than once and its operands haven't changed in between, the compiler avoids recomputing it by reusing the earlier result.

  Before:                   After:
  t1 = a + b               t1 = a + b
  t2 = c * d               t2 = c * d
  t3 = a + b      →        t3 = t1      ← reuse t1
  t4 = t1 + t3             t4 = t1 + t1
  t5 = c * d               t5 = t2      ← reuse t2

3. DEAD CODE ELIMINATION

Remove code that is never executed (unreachable) or computes values that are never used.

  // Unreachable code:
  x = 5;
  goto L1;
  y = x + 1;    ← dead: never reached
  L1: ...

  // Unused variable:
  int temp = computeExpensiveValue();
  // temp is never read → remove the computation
These three optimizations are often applied iteratively: constant propagation can expose new constant folding opportunities, which in turn can reveal dead code.
Q4.13 Explain with examples: (1) Constant Folding and Constant Propagation, (2) Strength Reduction, (3) Dead Code Elimination [Jan-May 2024 UE20CS353 Q4.b]
5 marks

1. CONSTANT FOLDING AND PROPAGATION — see Q4.12

2. STRENGTH REDUCTION

Replace expensive operations with cheaper equivalent ones:

  • Replace multiplication by a power of 2 with left shift: x * 4 → x << 2
  • Replace division by a power of 2 with right shift: x / 8 → x >> 3
  • In loops, replace multiplication by an induction variable with addition:
  // Before (multiplication every iteration):
  for (i = 0; i < n; i++)
      a[4*i] = 0;

  // After strength reduction:
  t = 0;
  for (i = 0; i < n; i++) {
      a[t] = 0;
      t = t + 4;    // cheaper: add instead of multiply
  }

3. DEAD CODE ELIMINATION — see Q4.12

Q4.14 What is code optimization? Explain with an example any two optimization techniques. [Jan-May 2024 UE19CS351 Q4.c / UE20CS353 Q4.c]
5 marks

DEFINITION

Code optimization is the process of transforming a program to make it execute more efficiently (faster, use less memory, or consume less energy) without changing the program's observable behavior (output).

Classification:
  • Machine-independent: Applied to intermediate code (e.g., constant folding, CSE, loop invariant code motion)
  • Machine-dependent: Target-specific (e.g., register allocation, peephole optimization)

EXAMPLE 1: LOOP INVARIANT CODE MOTION

Move computations that don't change inside a loop to outside the loop:

  Before:
  for (i = 0; i < n; i++)
      a[i] = b + c * 10;   // b+c*10 is loop-invariant

  After:
  t = b + c * 10;          // computed once
  for (i = 0; i < n; i++)
      a[i] = t;

EXAMPLE 2: CONSTANT FOLDING

  Before:  x = 2 * 3.14 * 5;
  After:   x = 31.4;           // computed at compile time
Q4.15 Apply loop strength reduction by induction variables, constant folding and dead code elimination to: i=0; L: i=i+1; t1=4*i; t2=a[t1]; if t2<v goto L [Jan-May 2024 UE21CS341B Q3.c]
3 marks

ORIGINAL CODE

  i = 0
  L: i = i + 1
     t1 = 4 * i
     t2 = a[t1]
     if t2 < v goto L

STEP 1: STRENGTH REDUCTION ON INDUCTION VARIABLE

Introduce t1 as an induction variable. Since t1 = 4*i and i increases by 1 each iteration, t1 increases by 4 each iteration:

  t1 = 4 * 0  → t1 = 0   (initial value, before loop)
  L: i = i + 1
     t1 = t1 + 4    ← strength reduced (add instead of multiply)
     t2 = a[t1]
     if t2 < v goto L

STEP 2: DEAD CODE ELIMINATION

Since i is only used to compute t1 and is not used elsewhere, i is now a dead variable. Remove it:

  t1 = 0
  L: t1 = t1 + 4
     t2 = a[t1]
     if t2 < v goto L
Q4.16 Reduce loop overhead from 100% to 50%: for(i=0; i<100; i++) g(); [Jan-May 2024 UE21CS341B Q3.c(i)]
2 marks

LOOP UNROLLING

The technique is called loop unrolling. By unrolling the loop body twice, we halve the number of loop control operations (increment, compare, branch):

  // Original: 100 iterations, 100 conditional branches
  for (i = 0; i < 100; i++)
      g();

  // After 2x unrolling: 50 iterations, 50 conditional branches
  for (i = 0; i < 100; i += 2) {
      g();
      g();
  }
Loop overhead reduced from 100 jumps/comparisons to 50 — a 50% reduction. The body is duplicated, so no calls are skipped. This increases code size but reduces loop control overhead.

§ 4.5 — Runtime Environments: Storage and Activation Records
Q4.23 Explain with neat diagrams: (1) Activation Record and (2) Activation Tree [Appears in multiple papers — 6 marks versions]
6 marks

1. ACTIVATION RECORD (Stack Frame)

An activation record (or stack frame) is a block of memory allocated on the runtime stack each time a procedure is called. It contains all the information needed during one execution (activation) of the procedure.

Fields explained:

  • Return value: Where the callee places the value to be returned
  • Actual parameters: Arguments from the caller
  • Control link (dynamic link): Points to the activation record of the caller
  • Access link (static link): Points to the enclosing scope's record (for nested procedures)
  • Saved machine status: Return address, saved registers
  • Local data: The procedure's local variables
  • Temporaries: Compiler-generated temporary values

2. ACTIVATION TREE

An activation tree represents all activations (calls) of all procedures during the execution of a program. Nodes are activations; a node for procedure p is a child of node for q if q called p.

Example: Consider the recursive factorial main→fact(3)→fact(2)→fact(1)→fact(0)

Key property: The call stack at any point during execution corresponds to the path from the root to the currently executing node in the activation tree.

6 marks

1 & 2. ACTIVATION RECORD AND TREE — See Q4.23

3. CALLING SEQUENCE

The calling sequence is the code generated to make a procedure call. It is divided between the caller and the callee:

Caller's tasks (before the call):

  1. Evaluate actual parameters and push them onto the stack
  2. Store the return address (address of instruction after the call)
  3. Jump to the callee's code

Callee's tasks (on entry):

  1. Save the return address and control link
  2. Save registers and machine status
  3. Allocate local variables and temporaries on the stack
  4. Set the access link (if needed for nested scopes)

Return sequence (callee and caller after call):

  1. Callee: place return value in designated location
  2. Callee: restore machine status and pop its activation record
  3. Callee: jump to return address
  4. Caller: retrieve the return value

Q4.27 Explain the different ways in which a procedure can access nonlocal data on a runtime stack [Appears in multiple papers — 4 marks]

METHODS TO ACCESS NONLOCAL DATA

When a procedure P needs data from an enclosing scope (not its own locals or globals), it uses one of these mechanisms:

1. ACCESS LINKS (Static Links)

Each activation record contains a pointer (the access link or static link) to the activation record of the immediately enclosing scope at the source-code level (not the caller).

  • To access a variable d levels up the nesting, follow d access links.
  • Time to access: O(nesting depth)

2. DISPLAYS

A display is a global array d[] where d[i] holds a pointer to the most recent activation record at nesting depth i.

  • Accessing a variable at nesting depth i: look up d[i] — takes O(1) time
  • Updating the display on each call/return adds slight overhead
  • Faster than access links for deep nesting
  d[0] → main's record
  d[1] → most recent depth-1 procedure's record
  d[2] → most recent depth-2 procedure's record
  ...

3. GLOBAL VARIABLES

Global (static) variables are stored at fixed memory addresses known at compile time. Any procedure can access them directly without any indirection through the stack.

4. HEAP ALLOCATION

For data that must outlive its creator, the data is allocated on the heap. Procedures access it through a pointer held in their activation record.

To summarize for 4 marks:

Displays: A global array of pointers indexed by nesting depth. display[i] always points to the currently active record at depth i. O(1) access to any nesting level. On call/return, the display must be updated, but access is faster than following multiple static links.

CALLING SEQUENCE

The calling sequence is the sequence of actions (and corresponding code) that must happen to transfer control from the caller to the callee and set up the callee's environment.

Division of responsibilities:

Callee's epilogue (on return)Caller's code (after return)

  ADD SP, 8       // pop arguments (caller-cleanup)
§ 4.6 — Issues in Code Generator Design
Q4.30 Explain in brief any four issues in the design of a code generator. [Appears in every paper — 4 marks]
4 marks

ISSUES IN CODE GENERATOR DESIGN

A code generator translates intermediate code (e.g., three-address code) into target machine code. The key design issues are:

1. INPUT TO THE CODE GENERATOR

The input is a sequence of intermediate-level instructions (three-address code, DAG, etc.) along with information from the symbol table (types, storage locations). The code generator assumes that the input is free of type errors (checked in semantic analysis).

2. TARGET PROGRAMS

The target can be absolute machine code (directly executable), relocatable machine code (requires linking), or assembly language. Assembly is easier to generate and debug; absolute code has the fastest startup.

3. MEMORY MANAGEMENT

Variables in the source program must be mapped to memory locations (or registers) in the target. The code generator uses the symbol table and the runtime environment (stack, heap, static) to do this mapping. Names in three-address code must be converted to actual addresses.

4. INSTRUCTION SELECTION

The code generator must choose instructions from the target ISA to implement each operation. A good selection produces efficient code. For example, a = a + 1 could be INC a (1 instruction) rather than MOV R, a; ADD R, #1; MOV a, R (3 instructions).

5. REGISTER ALLOCATION

Registers are fast but limited. The code generator must decide which values to keep in registers (register allocation) and which register to use for each value (register assignment). This is NP-complete in general; graph-coloring heuristics are used.

6. EVALUATION ORDER

The order in which computations are performed can affect the number of registers needed and the number of loads/stores. Some orderings allow intermediate results to remain in registers longer.

For a 4-mark answer: Pick any four from the above (Instruction Selection, Register Allocation, Memory Management, Evaluation Order are the most commonly expected).
Q4.31 Explain the tasks which are typically part of a sophisticated compiler's "code generation" phase. [May 2023 UE20CS353 Q5.b — 5 marks]
5 marks

CODE GENERATION PHASE TASKS

The code generation phase of a compiler is responsible for transforming optimized intermediate code into target machine (or assembly) code. Tasks include:

  1. Instruction Selection: Mapping intermediate code constructs to target machine instructions. The compiler chooses the most efficient instruction(s) for each operation.
  2. Register Allocation and Assignment: Deciding which program variables and temporaries should reside in registers. Register allocation assigns values to a set of registers; register assignment picks the specific register.
  3. Address Calculation: Computing the runtime address of each variable/array element (static, stack-relative, or heap-allocated).
  4. Instruction Scheduling: Reordering instructions to avoid pipeline stalls and to take advantage of instruction-level parallelism, without changing program semantics.
  5. Procedure Call/Return Code: Generating the calling and return sequences for function calls.
  6. Peephole Optimization: A local optimization pass that examines a small window of instructions and replaces them with more efficient equivalents (e.g., removing redundant loads/stores).
§ 4.7 — Target Code Generation: Simple Code Generator
Q4.32 Generate target code for: (1) x:=y+z, (2) z:=x*x, (3) y:=z, (4) x:=y+z. Two registers R1, R2. All variables live on exit. No temporaries. Arithmetic ops use registers only. [Appears in multiple papers — 6 marks]
6 marks

SIMPLE CODE GENERATOR ALGORITHM

For each instruction, use getReg() to find appropriate registers. Track register descriptor (which variable is in each register) and address descriptor (where each variable is stored: register, memory, or both).

INITIAL STATE

All variables (x, y, z) are in memory. R1 and R2 are free.

CODE GENERATION

Statement 1: x := y + z

  • Load y into R1: LD R1, y
  • Load z into R2: LD R2, z
  • Add: ADD R1, R1, R2 → R1 holds x's value
  • Address descriptor: x → R1 (only)

Statement 2: z := x * x

  • x is in R1. z can use R2 (R2 held z but z is now dead since we're redefining it)
  • MUL R2, R1, R1 → R2 holds z's value
  • Address descriptor: z → R2 (only)

Statement 3: y := z

  • z is in R2. y can reuse R2 (since z will be stored before R2 is reused)
  • Move: R2 holds both z and y. Address descriptor: y → R2 (only); z → R2 and memory (we need to store z since it's live on exit)
  • Actually: ST z, R2 first to preserve z if needed later... but y := z means y and z point to same value in R2.
  • Let y → R2 (R2 is the register for y). Since z is live on exit and currently only in R2, we store z: ST z, R2... but then we want to use R2 for y too. With only 2 registers this is tricky.
  • Simpler: MOV R1, R2 (if available) or just store y from R2: since y := z just copies z to y, and R2 already has z's value, R2 now describes y.

Statement 4: x := y + z

  • y is in R2 (or R1), z must come from memory or a register
  • Load z (from memory): LD R1, z
  • Compute: ADD R1, R2, R1 → R1 holds new x

COMPLETE TARGET CODE WITH REGISTER DESCRIPTORS

  Instruction      Code Generated         Register Desc   Address Desc
  ─────────────────────────────────────────────────────────────────────
  x := y + z       LD  R1, y             R1: y           y: R1,mem
                   LD  R2, z             R2: z           z: R2,mem
                   ADD R1, R1, R2        R1: x           x: R1

  z := x * x       MUL R2, R1, R1        R1: x, R2: z    z: R2

  LD  R2, z

  MUL R2, R1, R1     ; R2 = x*x = new z

Q4.35 Generate target code for static allocation: procedure p() with m=5, n=m*2, call q, halt; procedure q() with x=2*x, return. p at 100, q at 300; activation records at 400 and 600 respectively. [Jul 2023 UE20CS353 Q5.b]
5 marks

STATIC ALLOCATION

In static allocation, each procedure has a fixed activation record at a known address. Locations: p's code at 100, q's code at 300, p's activation record at 400 (m at 400, n at 404), q's activation record at 600 (x at 600).

  // Procedure p() — code starts at 100
  100: MOV  400, #5      ; m = 5
  108: MOV  R1, 400      ; load m into R1
  116: MUL  R1, R1, #2   ; R1 = m * 2
  124: MOV  404, R1      ; n = m*2
  132: MOV  600, #ret_addr ; save return address for q
  140: GOTO 300          ; call q (jump to q's code)
  148: HALT              ; halt (return address is 148)

  // Procedure q() — code starts at 300
  300: MOV  R1, 600      ; load x into R1
  308: MUL  R1, R1, #2   ; R1 = 2 * x
  316: MOV  600, R1      ; store back x = 2*x
  324: GOTO *ret_in_600  ; return: jump to return address
In static allocation, the return address is stored in the callee's activation record (at a fixed offset). There's no stack; each procedure has exactly one activation record at a compile-time-known address. This means no recursion is possible.
Q4.36 Generate target code for array accesses: x=a[i]; y=b[j]; a[i]=y; b[j]=x. Arrays a and b have 4-byte elements. Explain Index and Indirect addressing modes. [Dec 2024 Q4.b]

ADDRESSING MODES

Index Addressing: Address = base + index. Used for array access. Example: MOV R1, a(R2) means load from address a + R2, where a is the base address and R2 holds the byte offset.

Indirect Addressing: Address = value of register. Example: MOV R1, *R2 means load from the address contained in R2. Used for pointers.

TARGET CODE FOR ARRAY ACCESSES

Array element address: a[i] is at address base(a) + i * 4 (4 bytes per element).

  // x = a[i]
  MOV  R1, i          ; R1 = i
  MUL  R1, R1, #4     ; R1 = i * 4 (byte offset)
  MOV  R2, a(R1)      ; R2 = a[i]  (indexed: base a + R1)
  MOV  x, R2          ; x = a[i]

  // y = b[j]
  MOV  R1, j          ; R1 = j
  MUL  R1, R1, #4     ; R1 = j * 4
  MOV  R2, b(R1)      ; R2 = b[j]
  MOV  y, R2          ; y = b[j]

  // a[i] = y
  MOV  R2, y          ; R2 = y
  MOV  R1, i
  MUL  R1, R1, #4
  MOV  a(R1), R2      ; a[i] = y (store into indexed location)

  // b[j] = x
  MOV  R2, x          ; R2 = x
  MOV  R1, j
  MUL  R1, R1, #4
  MOV  b(R1), R2      ; b[j] = x
The indexed addressing instruction MOV R, a(Ri) computes the effective address as a + Ri and loads/stores from that address. This is exactly the pattern for array element access.

THREE-ADDRESS CODE FOR SWITCH

A switch statement is typically compiled using a sequence of conditional jumps (for small number of cases) or a jump table (for many cases). Here, with 2 cases:

  // switch(ch) with cases 1 and 2

  if ch != 1 goto L2     ; test case 1
  t1 = a + b             ; case 1 body: c = a + b
  c = t1
  goto Lend              ; break → jump past switch

  L2: if ch != 2 goto Ldefault  ; test case 2
  t2 = a - b             ; case 2 body: c = a - b
  c = t2
  goto Lend              ; break

  Ldefault:              ; default case
  ...                    ; default code

  Lend:                  ; after switch
For switch with many integer cases, compilers generate a jump table: an array where entry i holds the address of case i's code. The switch expression is used as an index into this array for O(1) dispatch.

DAG (Directed Acyclic Graph)

A DAG for an expression is like a syntax tree except that common subexpressions are represented by shared nodes rather than repeated subtrees. This avoids redundant computation and is the basis for Common Subexpression Elimination (CSE).

EXPRESSION ANALYSIS

  Let E = ((x+y) – ((x+y)*(x-y))) + ((x+y)*(x-y))
  Common subexpressions:
    A = x+y   (appears 3 times)
    B = x-y   (appears 2 times)
    C = A*B = (x+y)*(x-y)  (appears 2 times)

SYNTAX TREE (showing redundancy)

DAG (sharing common nodes)

In the DAG, x+y (node A) and (x+y)*(x-y) (node C) each have exactly one node with multiple incoming edges. This represents the sharing of common subexpressions.

Q4.44 Partition the basic block into CFG. Explain the Parser stack implementation of postfix SDTs. What is a dependency graph? [Jul 2023 UE20CS353 Q3.c, Q3.d]

PARSER STACK IMPLEMENTATION OF POSTFIX SDTs

In a bottom-up (LR) parser, semantic actions for a postfix SDT are executed when a production is reduced. The parser stack holds grammar symbols along with their synthesized attribute values.

  • When reducing A → XYZ, the attributes of X, Y, Z are on the stack
  • The semantic action computes A.val from X.val, Y.val, Z.val
  • A.val is pushed onto the stack in place of XYZ

Example for E → E₁ + T {E.val = E₁.val + T.val}:

  Stack before reduce:  ... | E (5) | + | T (3)
  Action: E.val = 5 + 3 = 8
  Stack after reduce:   ... | E (8)

DEPENDENCY GRAPH

A dependency graph represents the dependencies between attribute instances in a parse tree.

  • Each node is an attribute instance (a.b = attribute b at node a)
  • A directed edge from c.k to b.j means: to compute b.j, we need c.k first
  • A valid evaluation order must respect this graph (a topological sort)
  • If the graph has a cycle, the SDD has no valid evaluation order

Example for E → E₁ + T:

  E₁.val ──┐
            ├─→ E.val
  T.val  ──┘
Q4.45 Explain Register Allocation problem and the Graph Coloring approach to solve it.
concept

REGISTER ALLOCATION PROBLEM

The goal is to assign program variables and temporaries to a limited set of machine registers such that:

  • Two variables that are live at the same time (interfere) do not share a register
  • The number of registers used does not exceed the available registers k
  • If not possible (register pressure), some variables must be spilled to memory
Register allocation is NP-complete in general. The graph-coloring heuristic is the practical approach used in production compilers.

GRAPH COLORING APPROACH

Step 1: Build the Interference Graph

  • Nodes = variables/temporaries
  • Edge between u and v if they are simultaneously live (interfere) at some program point
  • Use liveness analysis to determine interference

Step 2: Color the Graph

  • Assign colors (= registers) to nodes such that no two adjacent nodes share a color
  • Try to use at most k colors (= k registers available)
  • If k-colorable: assign variables to registers directly
  • If not: spill a variable (remove a node, recompute), retry

Chaitin's Algorithm (simplified):

  1. Build interference graph
  2. If a node has < k neighbors, push it onto a stack and remove from graph
  3. Repeat step 2 until graph is empty or only high-degree nodes remain
  4. Pop nodes from stack and assign them colors not used by their already-colored neighbors
  5. If step 3 leaves high-degree nodes: spill one and restart

Q4.48 Explain Peephole Optimization with examples. What are the typical improvements made by peephole optimization? [General concept]
concept

PEEPHOLE OPTIMIZATION

Peephole optimization is a local optimization technique that examines a small window (the "peephole") of consecutive instructions and replaces them with a shorter or faster equivalent sequence.

TYPICAL IMPROVEMENTS

1. Redundant loads/stores:

  Before:  ST  R0, x    ; store R0 into x
           LD  R0, x    ; load x into R0 (redundant!)
  After:   ST  R0, x    ; the load is unnecessary

2. Unreachable code:

  Before:  GOTO L1
           x = 5        ← unreachable
           L1: ...
  After:   GOTO L1
           L1: ...      ; remove dead instruction

3. Flow-of-control optimizations:

  Before:  GOTO L1
           ...
  L1:      GOTO L2
  After:   GOTO L2      ; jump-to-jump optimization

4. Algebraic identities:

  x = x + 0  →  (remove)
  x = x * 1  →  (remove)
  x = x ** 2  →  x = x * x
Q4.49 Explain the Storage Organization of a running program (the different memory regions: code, static data, stack, heap). [General concept]
concept

MEMORY LAYOUT OF A RUNNING PROGRAM

Static Allocation: Global/static variables at fixed addresses. No runtime overhead. No recursion possible for locals.

Stack Allocation: Activation records for local variables and parameters. LIFO order matches procedure call/return. Supports recursion.

Heap Allocation: For data with dynamic lifetime (e.g., objects created with new/malloc). Programmer or GC responsible for deallocation.
