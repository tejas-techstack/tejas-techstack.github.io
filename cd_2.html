Q1 Explain in brief any two Error-Recovery Strategies. (Appears in multiple exams)
4 marks error
▼

1. Panic-Mode Recovery

The most widely used method. When an error is detected, the parser discards input symbols one at a time until a synchronizing token is found. Synchronizing tokens are usually delimiters such as semicolons (;), closing braces (}), or end-of-statement markers.

Algorithm: On error → discard tokens until a token in FOLLOW(A) is found → resume parsing at non-terminal A.

Advantage: Simple to implement. Disadvantage: Large portions of input may be skipped without reporting all errors.

2. Phrase-Level (Statement Mode) Recovery

When an error is discovered, the parser performs a local correction on the remaining input — it replaces a prefix of the remaining input with some string that allows the parser to continue. Typical corrections: inserting a missing semicolon, replacing a comma with a semicolon, or deleting an extraneous token.

Example: If if (x > 0 is missing closing parenthesis, phrase-level recovery inserts ) and continues.

Advantage: More errors reported in a single pass. Disadvantage: Incorrect corrections can cause cascading spurious errors.

3. Error Productions (bonus)

The grammar is augmented with productions that recognize known common mistakes. When such a production is used, an error is reported but parsing continues normally.

4. Global Correction (bonus)

The parser finds the globally minimum-cost sequence of insertions and deletions to convert the erroneous input into a valid string. Rarely used in practice due to high computational cost.

Q2 Explain in detail Panic-Mode and Phrase-Level (Statement Mode) Error-Recovery Strategies.
4–6 marks error
▼

Panic-Mode Recovery — Detailed

Panic-mode recovery is based on the idea that it is better to skip erroneous input and resume parsing than to get stuck.

  • A set of synchronizing tokens is defined for each non-terminal (typically FOLLOW set).
  • When an error is found, tokens are discarded until a synchronizing token appears.
  • The parser pops stack entries until it reaches a state that can shift the synchronizing token.
  • For example, in a language where ; ends statements, the parser can discard tokens until ; and then restart statement parsing.
Synchronizing tokens for non-terminal A ⊇ FOLLOW(A)
When error at A: discard input until token ∈ FOLLOW(A)

Pros: Simple, guaranteed not to loop forever. Cons: May skip large valid sections, missing real errors.

Phrase-Level Recovery — Detailed

The parser applies local corrections to the remaining input. The corrections are based on likely programmer mistakes.

  • Insertion: Insert a missing token (e.g., add ; after a statement).
  • Deletion: Remove an extra token (e.g., delete duplicate ;;)).
  • Substitution: Replace an incorrect token (e.g., replace , with ;).
  • Transposition: Swap two adjacent tokens.
Care must be taken that phrase-level corrections do not create infinite loops — always ensure the correction causes the parser to advance in the input.

Pros: More informative error messages, better recovery. Cons: Incorrect local corrections can cascade, producing misleading subsequent error messages.

Q3 Explain in detail Phrase-Level (Statement Mode) and Error Productions Error-Recovery Strategies.
4 marks error
▼

Phrase-Level Recovery

See Q2 above for full detail. The parser performs local token corrections (insert/delete/substitute) to allow parsing to continue. The correction is minimal — aimed at making the smallest change to allow the parser to proceed.

Error Productions

The compiler writer anticipates common programming errors and adds error productions to the grammar to handle them explicitly.

  • Each error production captures a known syntactic mistake.
  • When the error production fires, a diagnostic message is emitted and parsing continues normally.
Example: If forgetting = in assignment is common:
Normal: stmt → id = expr
Error prod: stmt → id expr { error("Missing ="); }

Another example: In Pascal, := vs = confusion:

assignment → variable = expression { error("Use := not ="); }

Advantage: Clean, predictable recovery for known mistakes. Disadvantage: Requires advance knowledge of common errors; grammar grows larger.

Q4 Explain with an example Panic-Mode and Phrase-Level (or Statement Mode) Error Recovery Strategies.
4 marks error
▼

Panic-Mode — With Example

Consider parsing a C-like language. Suppose the input is:

int x = ;   /* missing value after = */
int y = 5;

The parser encounters ; where it expects an expression. In panic mode:

  1. Error detected: expected expression, got ;
  2. Parser pops stack until it reaches a state that can accept a synchronizing token.
  3. Synchronizing tokens for statement = {;, }}
  4. ; is already visible → resumes after ;
  5. Now parses int y = 5; correctly.
One error reported: "syntax error near ';', expected expression"

Phrase-Level — With Example

Same input: int x = ;

  1. Error detected: expected expression, got ;
  2. Parser locally corrects by inserting a default value, say 0
  3. Now input is treated as int x = 0; — valid!
  4. Parsing continues without discarding tokens.
Message: "Missing expression after '='; assumed 0"

Another example: missing semicolon int x = 5 at end of line. Phrase-level inserts ; and continues.

Q5 What is the function of an Error Handling routine? What are the different ways in which error handling can be implemented by a compiler?
4 marks error
▼

Function of an Error Handler

When a parser encounters an error it must:

  • Detect the error clearly and as early as possible.
  • Report it with a precise, helpful diagnostic message (line number, what was expected, what was found).
  • Recover from the error and continue parsing to find more errors in a single pass.
  • Not slow down compilation of error-free programs significantly.

Error Recovery Methods

  1. Panic-Mode Recovery: Skip tokens until a synchronizing token (from FOLLOW set) is found. Simple but may skip valid code.
  2. Phrase-Level Recovery: Local correction — insert, delete, or substitute tokens near the error point to allow parsing to proceed.
  3. Error Productions: Augment grammar with productions that match known erroneous constructs. Fire the production with an error message.
  4. Global Correction: Compute minimum edit distance between the erroneous string and the nearest correct string. Theoretically optimal but expensive; rarely used in production compilers.
Most modern compilers use a combination of panic-mode and phrase-level recovery, tuned for the specific language being compiled.
Q6 Provide an example where the lexer requires help from other phases of the compiler to identify the correct role (token name) of a currently matched lexeme.
2 marks error
▼

Context-Dependent Tokenization

A classic example comes from C/C++ typedef:

typedef int myInt;
myInt x;   // Is 'myInt' a type-name or an identifier?

The lexer cannot determine whether myInt is a type-name token or an ordinary identifier token without consulting the symbol table maintained by the semantic phase. The parser (or semantic analyzer) must inform the lexer about type definitions so the lexer can return the correct token type.

Another Example — Fortran

DO 10 I = 1.10   /* assignment to variable DO10I */
DO 10 I = 1, 10  /* DO loop */

Here the presence of a comma vs. period determines whether DO is a keyword or the start of a variable name. The lexer needs to look ahead (or the parser must feed context back) to correctly tokenize.

This demonstrates that lexical and syntactic analysis are not always cleanly separable — scanners sometimes need parser feedback for ambiguous constructs.
Q7 With simple example explain: (i) Left-recursive grammar (ii) Leftmost derivation (iii) Ambiguous grammar (iv) LR(0) item
4 marks misc
▼

(i) Left-Recursive Grammar

A grammar is left-recursive if there exists a non-terminal A such that A ⇒⁺ Aα for some string α.

Example: E → E + T | T (directly left-recursive, E derives Eα)

Left recursion causes top-down parsers (LL) to loop infinitely. Bottom-up (LR) parsers handle it fine.

(ii) Leftmost Derivation

At each step, the leftmost non-terminal in the sentential form is replaced.

Grammar: E → E+E | id
Leftmost: E ⇒ E+E ⇒ id+E ⇒ id+id

Leftmost derivations correspond to top-down (LL) parsing.

(iii) Ambiguous Grammar

A grammar is ambiguous if some string in the language has more than one parse tree (equivalently, more than one leftmost or rightmost derivation).

E → E + E | E * E | id
For "id + id * id": two parse trees exist (add first, or multiply first).

(iv) LR(0) Item

An LR(0) item is a grammar production with a dot (•) placed somewhere in its body, indicating how far parsing has progressed.

Production: A → αβ
Items: A → •αβ, A → α•β, A → αβ•

Example: E → E • + T (we have seen E, expecting + then T)

A dot at the end (A → αβ•) is a complete item, indicating a reduce action.

Q14 Eliminate left recursion from: A→Bxy|x, B→CD, C→A|c, D→d (indirect left recursion).
3 marks ll1
▼

Identify Indirect Left Recursion

A→Bxy, B→CD, C→A → so A⇒Bxy⇒CDxy⇒Axy → A ⇒⁺ Axy — left-recursive!

Algorithm (order A, B, C, D)

Step 1: Process A: no immediate left recursion.

Step 2: Process B: B→CD (C comes before B in order, substitute C→A|c):

B → AD | cD

Step 3: Process C: C→A|c. A appears before C. Substitute A→Bxy|x:

C → Bxyy... wait, A→Bxy|x so C→Bxy|x|c

Now B appears before C — substitute B→AD|cD:

C → ADxy | cDxy | x | c

Now remove immediate left recursion from A using substituted form. A→Bxy|x and B→AD|cD:

A → ADxy | cDxy | x | x ... A → ADxy | cDxy | x

A→ADxy | cDxy | x — A is immediately left-recursive in A→ADxy.

Eliminate: let α = Dxy, β₁=cDxy, β₂=x

A → cDxy A' | x A'
A' → Dxy A' | ε

Final grammar (no left recursion):

A  → cDxy A' | x A'
A' → Dxy A' | ε
B  → AD | cD
C  → ADxy | cDxy | x | c
D  → d

Left Factoring

All three productions start with 'a'. Factor out 'a':

A → a A'
A' → AB | Bc | Ac

Now A' has two productions starting with A (AB, Ac) and one starting with B (Bc). Factor A-prefix:

A  → a A'
A' → A A'' | B c
A'' → B | c

This is the fully left-factored form with no common prefixes remaining.

Q17 Explain in brief Recursive-Descent Parsing (RDP).
2 marks ll1
▼

Recursive-Descent Parsing

A top-down parsing technique that consists of a set of recursive procedures, one per non-terminal. Each procedure parses the portion of the input that matches its non-terminal.

For grammar E → T + E | T, T → int:

function E():
  T()
  if lookahead == '+':
    match('+')
    E()

function T():
  match('int')
  • Uses lookahead token to choose among alternatives (1 lookahead = LL(1)).
  • May use backtracking when there are multiple alternatives (general RDP).
  • Simple to hand-code from a grammar.
  • Cannot handle left-recursive grammars directly — left recursion must be eliminated first.

Q21 Grammar: S→AaAb | BbBa, A→ε, B→ε — LR(0) automata, SLR table, is it SLR?

Augmented Grammar

S' → S
S  → AaAb | BbBa
A  → ε
B  → ε

Key Observation

Since A→ε and B→ε, both appear at positions before terminals. When we see 'a' on the input, we need to decide whether we're in the S→AaAb production (where A→ε) or in S→BbBa. The LR(0) automaton state after reading nothing will contain both A→•ε and B→•ε, both of which immediately complete (dot at end).

LR(0) States (simplified)

I0: S'→•S, S→•AaAb, S→•BbBa, A→•ε, B→•ε, A→ε•, B→ε•
     (A→ε• and B→ε• are complete items — reduce items)

In I0 we must reduce A→ε or B→ε. On token 'a': we need A→ε (to proceed S→AaAb). On token 'b': we need B→ε (to proceed S→BbBa).

FOLLOW Sets

FOLLOW(A) = {a, b} (from S→AaAb: A followed by a; from S→AaAb: second A followed by b)

FOLLOW(B) = {b, a} (from S→BbBa: B followed by b; second B followed by a)

In I0: A→ε• (reduce on FOLLOW(A)={a,b}) and B→ε• (reduce on FOLLOW(B)={b,a}) — both have {a,b} in follow. Reduce-reduce conflict on both a and b!

The grammar is NOT SLR(1). It requires CLR(1) (canonical LR) to resolve because the necessary context (whether we're in the AaAb or BbBa rule) cannot be determined from SLR(1) lookaheads alone.

Q29 Explain in brief Canonical LR(1) Items. Why is LR(1)/CLR(1) parser so powerful?
4 marks clr
▼

Canonical LR(1) Items

A LR(1) item is an LR(0) item augmented with a lookahead terminal:

[ A → α • β, a ]

where a is the lookahead. The item means: we have recognized α; we expect to see β followed eventually by 'a'.

  • When β ≠ ε (dot not at end): the lookahead has no effect; shift proceeds as in LR(0).
  • When β = ε (dot at end, i.e., [A→α•, a]): reduce A→α only when the next token is 'a'.
Example:
[S → L • = R, $] means: saw L, expecting = then R, and $ follows the whole S
[S → L •, =] means: L is complete, reduce S→L only if next token is =

CLOSURE for LR(1)

For item [A→α•Bβ, a], add [B→•γ, b] for each B-production and each b ∈ FIRST(βa).

Why CLR(1) is Most Powerful

  • Lookaheads are computed precisely from the grammatical context, not approximated (like FOLLOW sets in SLR).
  • Can distinguish cases where SLR and LALR produce spurious conflicts.
  • Recognizes the largest class of grammars among deterministic LR parsers: every unambiguous grammar that can be parsed deterministically bottom-up is CLR(1).
  • FOLLOW sets (SLR) are global and may include tokens that can never actually follow A in a specific context; LR(1) lookaheads are exact.
Disadvantage: CLR(1) can have many more states than LALR(1) for the same grammar, making it less practical. LALR(1) achieves almost the same power with fewer states.

LALR(1) — Lookahead LR(1)

LALR(1) is constructed by:

  1. Building the full CLR(1) automaton.
  2. Merging states that have the same core (same LR(0) items, ignoring lookaheads).
  3. The merged lookaheads = union of lookaheads from merged states.
If CLR has states {[A→α•,a],[B→β•,c]} and {[A→α•,b],[B→β•,d]}
LALR merges them to: {[A→α•,a/b],[B→β•,c/d]}

StatesMore (exact)Fewer (merged)
LookaheadsPrecise per-stateUnion of merged
ConflictsNone for LR grammarsMay introduce R-R conflicts
Practical useRarely (too large)yacc, bison, most tools

Same number of states as SLR but much more precise lookaheads. Most programming language grammars that are CLR(1) are also LALR(1). The table size is manageable for realistic languages.

Q33 What is viable prefix? Explain its role in LR parsing.
3 marks clr
▼

Viable Prefix

A viable prefix is any prefix of a right-sentential form that does not extend past the right end of the rightmost handle.

If γ = αβw is a right-sentential form with handle β at position α,
then any prefix of αβ is a viable prefix.
  • Viable prefixes represent what appears on the LR parser stack at any point during parsing.
  • They are recognized by a finite automaton (the LR(0) automaton).
  • The LR parsing stack always contains a viable prefix of some right-sentential form.

Role in LR Parsing

  • The LR(0) items encode all possible viable prefixes.
  • Each state in the LR automaton corresponds to a set of items representing a particular viable prefix seen so far.
  • The parser performs an error-free parse only when the current stack content is a viable prefix.
  • Error is detected as soon as a string that is not a viable prefix is pushed.

Q34 Write short notes on: (1) Types of Attributes (2) Types of Syntax Directed Translations (SDTs)
8 marks attributes
▼

1. Types of Attributes

Attributes are values associated with grammar symbols (terminals and non-terminals). They store information computed during parsing.

Synthesized Attributes:

  • Value is computed from the attributes of the node's children in the parse tree.
  • Information flows bottom-up (from leaves to root).
  • All attributes of terminals are synthesized (their values come from the lexer).
E → E₁ + T { E.val = E₁.val + T.val } ← val is synthesized

Inherited Attributes:

  • Value is computed from the attributes of the node's parent and siblings.
  • Information flows top-down or across siblings.
  • Used to pass context downward (e.g., type information to variable declarations).
D → T L { L.type = T.type } ← type is inherited by L from T

2. Types of Syntax Directed Translations

S-Attributed SDD:

  • Uses only synthesized attributes.
  • Can be evaluated with a single bottom-up pass.
  • Implemented easily on top of LR parsers by computing attributes on the parser stack during reductions.
  • Simpler and more restrictive.

L-Attributed SDD:

  • Can use both synthesized and inherited attributes, with the restriction that inherited attributes of the j-th symbol on the RHS can only depend on:
    • Inherited attributes of the head (LHS non-terminal), and
    • Attributes of symbols to the left of the j-th symbol in the production.
  • Can be evaluated in a single left-to-right pass over the parse tree.
  • Works naturally with LL parsers and with recursive-descent parsers.
Every S-attributed SDD is also L-attributed.
Not every L-attributed SDD is S-attributed.

Postfix SDT: Semantic actions placed after all RHS symbols — directly computed during LR parsing.

General SDT: Actions placed anywhere in the RHS; may require multiple passes or translation schemes.

Q37 What is an inherited attribute? (2 marks). What does 'L' stand for in L-attributed SDD? If not L-attributed, can it ever be evaluated? (2 marks)
4 marks attributes
▼

Inherited Attribute

An attribute b of a node N is inherited if its value is defined by a semantic rule at the parent of N (or by a sibling to the left of N). Inherited attributes allow information to flow from the root downward through the parse tree or across siblings from left to right.

Example: In D→TL with rule L.type=T.type, L.type is inherited from T's synthesized attribute via the parent D.

L in L-Attributed

L stands for "Left-to-right" — the attributes can be evaluated in a single left-to-right, depth-first traversal of the parse tree.

Formally, in an L-attributed SDD, every inherited attribute of the j-th grammar symbol Xⱼ on the RHS of A→X₁X₂…Xₙ depends only on:

  • Attributes of symbols X₁,…,Xⱼ₋₁ (to the left of Xⱼ), and
  • Inherited attributes of A (the head)

Can Non-L-Attributed SDDs Be Evaluated?

Yes, but not in a single left-to-right pass. Options include:

  • Multiple passes over the parse tree.
  • Build the parse tree first, then evaluate in a topological order of the dependency graph.
  • Transform the grammar to make it L-attributed (sometimes possible).

The dependency graph of the attributes must be acyclic for any evaluation order to exist.

Q38 Explain the procedure to convert any SDD to an SDT scheme.
2 marks attributes

SDD → SDT Conversion

An SDT (Syntax-Directed Translation scheme) embeds semantic actions directly into the production rule's right-hand side.

Rules for Conversion

  1. S-attributed SDD → Postfix SDT: Place each action at the end of the corresponding production's RHS. The action computes the LHS non-terminal's synthesized attribute.
  2. L-attributed SDD → SDT:
    • Actions computing inherited attributes of the (j+1)-th symbol are placed just before the (j+1)-th symbol.
    • Actions computing synthesized attributes of the LHS are placed at the end of the RHS.
SDD: A → B C { A.s = B.s + C.s; C.i = B.s }
SDT: A → B { C.i = B.s; } C { A.s = B.s + C.s; }
General SDDs (non-L-attributed) may need multiple passes and cannot always be directly converted to a single-pass SDT.

What is a DAG?

A Directed Acyclic Graph (DAG) is a compact representation of an expression where each unique subexpression has exactly one node (shared node). If a subexpression appears multiple times, all occurrences point to the same node rather than creating duplicate subtrees.

DAGs are more compact than syntax trees and allow common subexpression identification for optimization.

Subexpressions: x, y, (x+y)=A, (x-y)=B, A*B, A-(A*B), A*B [same!], result.

DAG (Shared Nodes)

The DAG has fewer nodes than the syntax tree. Nodes for (x+y) and (x-y) are each computed once and shared by multiple parents.

Rules for L-Attributed SDD → SDT Conversion

  1. Inherited attribute of Xⱼ: Place the action that computes the inherited attribute of the (j+1)-th grammar symbol immediately before the (j+1)-th grammar symbol Xⱼ₊₁ in the production body.
  2. Synthesized attribute of LHS: Place the action that computes the synthesized attribute of the head (LHS non-terminal A) at the very end of the production body.
  3. Multiple actions: If multiple attributes of the same symbol must be computed, group all relevant assignments into one action block placed at the correct position.
  4. Ordering: Within a single action block, the order must reflect dependencies — compute attributes in dependency order.
SDD: A → B C { B.i = A.i; C.i = B.s; A.s = C.s }
SDT: A → { B.i = A.i; } B { C.i = B.s; } C { A.s = C.s; }
The inherited attribute B.i (depends on A.i) is placed before B. The inherited attribute C.i (depends on B.s) is placed after B and before C. The synthesized A.s is placed at the end.
