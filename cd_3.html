1. Types of Attributes

In a Syntax-Directed Definition (SDD), every grammar symbol has a set of attributes — values associated with nodes in a parse tree that encode semantic information.

Synthesized Attributes

  • The value of an attribute at a node is computed from the attribute values of its children.
  • Information flows bottom-up in the parse tree.
  • Formal rule: For production A → α, a synthesized attribute of A depends only on attributes of symbols in α.
  • Example: In arithmetic expression evaluation, E.val is synthesized:
    E → E₁ + T { E.val = E₁.val + T.val }

Inherited Attributes

  • The value of an attribute at a node is computed from the attribute values of its parent and/or siblings.
  • Information flows top-down (or sideways) in the parse tree.
  • Formal rule: For production A → α B β, an inherited attribute of B may depend on attributes of A and other symbols in αβ.
  • Example: Passing type information down in a declaration:
    L → L₁, id { L₁.inh = L.inh }
Note: The start symbol cannot have inherited attributes (in standard usage). An SDD with only synthesized attributes is called S-attributed.

2. Types of Syntax-Directed Translations (SDTs)

An SDT is an SDD with embedded semantic actions (program fragments) placed at various positions inside grammar productions.

S-Attributed SDT (S-Attributed SDD)

  • Only synthesized attributes are used.
  • Actions are placed at the right end of the production body.
  • Can be evaluated during a single bottom-up (LR) parse.
  • Actions are executed during reductions.
  • Example:
    E → E₁ + T { E.val = E₁.val + T.val }

L-Attributed SDT (L-Attributed SDD)

  • Allows both synthesized attributes and inherited attributes, with a constraint: inherited attributes of a symbol Xⱼ in the body X₁X₂…Xₙ can only depend on:
    1. Inherited attributes of the head A
    2. Attributes (synthesized or inherited) of X₁…Xⱼ₋₁ (symbols to the left)
  • Actions can be placed anywhere in the production body (between any two symbols).
  • Can be evaluated during a top-down (LL) parse using a depth-first, left-to-right traversal.
  • Every S-attributed SDD is also L-attributed.
3a Explain the following in detail: (i) Synthesized Attributes and Inherited Attributes (ii) L-attributed SDT and S-attributed SDT 8 M

Synthesized Attributes

A synthesized attribute of a node N in a parse tree is defined by a semantic rule associated with the production at N. The value depends solely on attribute values at the children of N.

  • Information propagates bottom-up.
  • Can be evaluated in a single bottom-up traversal (LR parsing).
Production:  E → E₁ + T
Rule:        E.val = E₁.val + T.val   ← depends on children

Inherited Attributes

An inherited attribute of a node N is defined by semantic rules associated with the production at the parent of N. The value depends on attributes at the parent or siblings.

  • Information propagates top-down or sideways.
  • Used to pass context (e.g., type information) down the tree.
Production:  D  → T L
             L  → L₁, id | id
Rule:        L.inh = T.type        ← T's synthesized flows to L as inherited
             L₁.inh = L.inh
             addtype(id.entry, L.inh)

S-attributed SDD

  • Uses only synthesized attributes.
  • Semantic actions placed at the right end of each production.
  • Evaluation order: bottom-up post-order traversal.
  • Directly implementable with an LR parser using a value stack.

L-attributed SDD

  • Each inherited attribute of Xⱼ in A → X₁X₂…Xₙ depends only on:
    1. Inherited attributes of A
    2. Any attribute of X₁, …, Xⱼ₋₁ (left siblings)
  • Actions can appear anywhere in the body.
  • Suitable for top-down (predictive / LL) parsers.
  • Dependency graph has no cycles when following left-to-right, top-down traversal.
Key relationship: Every S-attributed SDD is also L-attributed, but not vice versa.

(ii) Dependency Graph

A dependency graph depicts the interdependencies between attribute instances in a parse tree. There is a directed edge from attribute instance b to attribute instance c (written b → c) if the value of c depends on the value of b.

  • Nodes represent attribute instances at parse-tree nodes.
  • Edges represent data dependencies (a → b means "a must be computed before b").
  • A topological sort of the dependency graph gives a valid evaluation order.
  • If the dependency graph has a cycle, the SDD cannot be evaluated.
A dependency graph with no cycles is called a DAG (Directed Acyclic Graph). Only cycle-free SDDs have well-defined evaluation orders.

Short-Circuit Evaluation of B₁ ‖ B₂

For B₁ || B₂: if B₁ is true, the whole expression is true (no need to evaluate B₂). Only if B₁ is false do we evaluate B₂.

SDD

ProductionSemantic Rules
C → B₁ ‖ B₂ L = new();
B₁.true = C.true;
B₁.false = L;
B₂.true = C.true;
B₂.false = C.false;
C.code = B₁.code ‖ label(L) ‖ B₂.code

Generated Three-Address Code

      <B₁.code>
      if B₁ goto C.true    // short-circuit: B₁ true → whole OR true
      goto L               // B₁ false → evaluate B₂
L:    <B₂.code>
      if B₂ goto C.true    // B₂ true
      goto C.false         // both false → overall false

Explanation

  • C.true and C.false are inherited labels from the enclosing context (e.g., if-statement).
  • L is a fresh label inserted between B₁ and B₂ evaluation.
  • If B₁ is true → jump directly to C.true (short circuit).
  • If B₁ is false → fall through to L and evaluate B₂.

SDT Schemes, Postfix & L-Attributed Implementation
3c Explain parser stack implementation of postfix SDTs with the help of a diagram. 5 M

Postfix SDT

A postfix SDT (S-attributed) has all semantic actions placed at the rightmost end of every production. This allows actions to be executed during LR parsing at reduction time.

Parser Stack Implementation

The LR parser maintains a stack of (state, value) pairs. When a reduction occurs:

  1. Pop the right-hand side symbols and their attributes from the value stack.
  2. Execute the semantic action to compute the left-hand side attribute.
  3. Push the new value onto the stack.

Example

Production:  E → E₁ + T  { E.val = E₁.val + T.val }

During LR parse of 3+5:
Stack before reduce:   [E.val=3, +, T.val=5]
Action:                E.val = 3 + 5 = 8
Stack after reduce:    [E.val=8]

Key Properties

  • Works only for S-attributed SDDs (synthesized attributes only).
  • At reduce time, the values of the body's symbols are at known positions relative to the top of the value stack.
  • If production has body of length n, the values are at val[top-n+1] through val[top].

Converting to Bottom-Up SDT

For LR parsing, actions with mid-production positions are handled by introducing marker non-terminals. Each marker Mᵢ has an ε-production with the embedded action.

S  → if (C) M₁ S₁ else M₂ S₂
       { S.code = C.code ‖ label(M₁.label) ‖ S₁.code 
                         ‖ label(M₂.label) ‖ S₂.code }

M₁ → ε  { M₁.label = new();
           backpatch(C.truelist, M₁.label); }

M₂ → ε  { M₂.label = new();
           emit('goto' S.next);
           backpatch(C.falselist, M₂.label); }

Simplified Version (using emit directly)

S  → if (C) M S₁ else M S₂
       { S.code generated via backpatching }

M  → ε  { M.quad = nextquad;
           emit('goto' _); }  // placeholder
The key technique: any mid-production action is moved into a new marker non-terminal M → ε with that action. LR parsers reduce M → ε at the right point in parsing.

4a What is a DAG? Draw a syntax tree and DAG for: ((x+y) – ((x+y)*(x–y))) + ((x+y)*(x–y)) 7 M

What is a DAG?

A Directed Acyclic Graph (DAG) for an expression is like a syntax tree except that nodes representing common subexpressions are shared (have multiple parents) rather than duplicated. This compact representation:

  • Identifies all common subexpressions.
  • Reduces redundant computations.
  • Is used in value-numbering optimization.

Identifying Common Subexpressions

In the expression ((x+y) – ((x+y)*(x–y))) + ((x+y)*(x–y)):

  • x+y — appears 3 times → node A
  • x-y — appears 2 times → node B
  • (x+y)*(x-y) — appears 2 times → node C = A * B
  • (x+y) - C → node D = A - C
  • root = D + C

DAG Diagram

In the DAG, nodes A (x+y), B (x-y), and C (A*B) are each computed once and reused. This eliminates redundant computation of common subexpressions.

(i) Quadruples

A quadruple is a record structure with four fields: (op, arg1, arg2, result). Each three-address instruction is represented as one quadruple.

Instructionoparg1arg2result
x = y + z+yzx
x = -yuminusy—x
x = y=y—x
param xparamx——
call p, ncallpn—
if x<y goto L<xyL

Advantages: Easy to access and reorder; result field names exactly where the result is stored.

Disadvantage: Requires many temporary variable names; the symbol table can become large.

Example: Quadruple representation of param and call

param x
param y
t1 = call add, 2
sum = t1

Quadruples:
(0): (param, x, —, —)
(1): (param, y, —, —)
(2): (call, add, 2, t1)
(3): (=, t1, —, sum)

(ii) Static Single Assignment (SSA) Form

SSA form is an intermediate representation in which every variable is assigned exactly once. When a variable has multiple definitions (e.g., in different branches), they are renamed with subscripts.

Key properties:

  • Each variable name corresponds to exactly one definition.
  • Uses special φ-functions (phi-functions) at join points (where control flow merges).
  • x₃ = φ(x₁, x₂) means x₃ gets the value of x₁ if coming from one predecessor, or x₂ from the other.

3d What is a dependency graph? Explain with example. 4 M

Dependency Graph

A dependency graph is a directed graph that depicts the flow of information between attribute instances in a parse tree. It is used to determine a valid evaluation order for the attributes in an SDD.

Construction Rules

  • For each parse tree node, create a node in the dependency graph for each attribute instance at that node.
  • For each semantic rule b = f(c₁, c₂, ..., cₙ), add edges cᵢ → b for each i (each cᵢ must be computed before b).

Example

For the declaration grammar with D→TL, T→int, L→L₁,id and input int a, b:

Topological Order (evaluation order)

T.type → L.inh → L₁.inh → addtype(a, L.inh), addtype(b, L₁.inh)

3b What is an inherited attribute? What does L stand for in L-attributed SDD? If the SDD is not L-attributed, does that mean it can never be evaluated? 6 M

Inherited Attribute

An inherited attribute of a node N in a parse tree is computed using attribute values of N's parent, siblings, or N's own other attributes. Information flows from the root toward the leaves (top-down), or from left siblings to the current node.

Example: L.inh = T.type in D → T L — L's type is inherited from T (its sibling) via D.

L in L-attributed SDD

L stands for "Left" — referring to the constraint that an inherited attribute of a symbol can only depend on symbols to its left in the production, or on the head's inherited attributes.

Formally, for production A → X₁ X₂ … Xₙ, inherited attribute of Xⱼ depends only on:

  1. Inherited attributes of A (head)
  2. Any attribute of X₁, X₂, …, Xⱼ₋₁ (symbols to the left)

This ensures evaluation can proceed left-to-right, top-down — matching depth-first traversal of a top-down parser.

Can non-L-attributed SDDs be evaluated?

Yes, they can still be evaluated, but not during a standard single-pass parse. Options:

  • Build the full parse tree first, then evaluate using a topological sort of the dependency graph.
  • If the dependency graph has no cycles, any SDD can be evaluated via topological order.
  • The restriction to L-attributed is a practical implementation constraint (one-pass evaluation), not a theoretical impossibility.
An SDD where the dependency graph has cycles cannot be evaluated at all — it represents circular dependencies.

Converting SDD to SDT

An SDT (Syntax-Directed Translation scheme) embeds semantic actions directly into the grammar production bodies. The conversion rules depend on whether the SDD is S-attributed or L-attributed:

For S-attributed SDDs (only synthesized attributes)

  1. Place all semantic actions at the right end of the production body.
  2. The action is executed when the production is reduced.
E → E₁ + T  { E.val = E₁.val + T.val }  // at the end

For L-attributed SDDs (synthesized + inherited)

  1. Actions that compute inherited attributes for grammar symbol Xⱼ are placed immediately before Xⱼ in the production body.
  2. Actions that compute synthesized attributes for the head A are placed at the right end.
D → { L.inh = T.type; } T L  // action before L to set inherited attr
                { /* final synthesized attrs here */ }

2d Explain Inherited Attribute and L-attributed SDT with suitable examples. 5 M

Inherited Attribute

An inherited attribute of a node N is defined using attribute values from N's parent or left siblings in the parse tree. It carries context into subtrees.

Example: Type propagation in declarations

Grammar:  D → T L,  T → int | float,  L → L,id | id
Rule:     D → T L  { L.inh = T.type }  ← T.type inherited by L
          L → id   { addtype(id.entry, L.inh) }

For "float a, b":
  T.type = float (synthesized)
  L.inh = float (inherited from T)
  addtype(a, float), addtype(b, float)

L-attributed SDT

In an L-attributed SDD, each inherited attribute of Xⱼ in A → X₁…Xₙ depends on:

  1. Inherited attributes of head A
  2. Any attribute of X₁, …, Xⱼ₋₁ (left of Xⱼ)

Example: Array type calculation

T  → BC         { T.type = array(C.num, B.type) }
B  → int        { B.type = integer }
C  → [num]      { C.num = num.val }

This is L-attributed: C.num is synthesized from num (its child), and T.type is synthesized from B.type and C.num. No cycles, no right-to-left dependencies.

L-attributed SDDs allow implementation during a single left-to-right, top-down parse. Actions for inherited attributes are placed before the respective non-terminal in the SDT.

2c Explain Inherited Attribute and L-attributed SDT with suitable examples. [second variant] 4 M

See full answer in Q37 above.

Quick Summary

Inherited Attribute

  • Defined at a node using its parent's or siblings' attributes.
  • Carries context from outer to inner parts of the parse tree (top-down flow).
  • Example: L.inh = T.type to propagate type down a variable list.

L-attributed SDT

  • L = "Left" dependency only.
  • Each Xⱼ's inherited attribute depends only on head's inherited attrs and left siblings' attributes.
  • Can be evaluated in a single left-to-right, top-down parse.
  • Every S-attributed SDD is also L-attributed.
3a Define: (1) Synthesized attributes (2) L-attributed SDD (3) S-attributed SDD 3 M

1. Synthesized Attributes

An attribute whose value at a parse-tree node N is computed from attribute values at N's children. Information flows bottom-up. Example: E.val = E₁.val + T.val.

2. L-attributed SDD

An SDD where each inherited attribute of Xⱼ in A → X₁…Xₙ depends only on A's inherited attributes and attributes of X₁,…,Xⱼ₋₁ (symbols to the left). Supports evaluation during left-to-right, top-down parsing. Includes all S-attributed SDDs.

3. S-attributed SDD

An SDD that uses only synthesized attributes. Actions are placed at the right end of productions. Directly implementable with an LR parser using a value stack. Simplest and most common type.

4d Define use[B] and def[B]. Calculate use[B], def[B] for all basic blocks in the given flow graph (B1:e=1; B2:if(x>0); B3:z=e*e; B4:return x; B5:y=e*x; B6:if x&1; B7:e=z; B8:e=y) 5 M

Definitions

  • use[B]: The set of variables that are used in block B before any definition of them in B (i.e., variables whose values from outside B are needed).
  • def[B]: The set of variables that are defined (assigned) in block B before any use of them in B.

A name in TAC (an "address") can be one of:

TypeDescriptionExample

Instruction Categories

  • Arithmetic/logic: t1 = a + b, t2 = a & b

  • Function call: param x; call f, n; return y
